## What this document is

This is the reference document for building `content.enjoy.hr` — a purpose-built
admin interface for the Enjoy Croatia editorial pipeline. The pipeline itself
(n8n workflows, Airtable, NeuronWriter) is already built and running. This site
is a UI layer on top of it.

Read this document fully before writing any code. Everything the admin site needs
to know about the backend is here.

---

## 1. System overview

The pipeline is a status-driven editorial workflow that produces SEO articles
for enjoy.hr, a Croatian tourism website for English-speaking visitors.

```
Airtable (state store)
    ↕
n8n (automation — 5 workflows)
    ↕
Claude API (writing, revision, metadata)
    ↕
WordPress REST API (publishing)
```

The admin site connects to:
- **Airtable API** — reads and writes article state (the source of truth)
- **n8n webhooks** — triggers workflow actions on demand
- **WordPress REST API** — optional direct integration for future use

The admin site does NOT replace n8n. It is a control panel that reads Airtable
state and triggers n8n where needed. All actual automation logic stays in n8n.

## 4. Status flow — complete reference

These are the exact status string values used in Airtable. Case-sensitive.
n8n polls for specific statuses — do not modify these strings.

```
backlog
  → brief-ready          (NLP brief pasted, WF1 fires automatically)
  → draft-ready          (WF1/WF2 complete, ready for editor review)
  → needs-revision       (editor requests changes, WF2 fires automatically)
  → needs-enrichment     (needs web-searched facts, WF3 fires — v2)
  → awaiting-fact-check  (after enrichment, editor must verify sources)
  → approved             (content approved, ready for images)
  → awaiting-images      (NEW — editor has written image brief, student notified)
  → images-submitted     (NEW — student uploaded candidates, editor notified)
  → image-approved       (editor selected image, WF5 fires automatically)
  → wp-draft             (WordPress draft created)
  → published            (live on enjoy.hr)
  → on-hold              (paused)
  → error                (pipeline failure — read Error Log)
  → rejected             (article killed)
```

**Two new statuses added for the image contributor workflow:**
- `awaiting-images` — editor has approved content and written the image brief.
  The student sees this article in their queue.
- `images-submitted` — student has uploaded candidates. Editor sees this
  in their "needs attention" list.

These need to be added to the Airtable Status field options before the admin
site can use them.

**Two new fields needed in Airtable Articles table:**
- `Image Brief` — Long text
- `Image Source Notes` — Long text

## 5. n8n workflows — what they do and how to trigger them

All workflows are hosted on the same n8n instance. They run automatically by
polling Airtable every 5 minutes, but some can also be triggered on demand
via webhooks from the admin site.

### WF1 — Draft Generation
- **Trigger:** Airtable status = `brief-ready` (polls every 5 min)
- **What it does:** Fetches First Draft prompt from Templates table, calls
  Claude API, saves draft HTML + plain text, sets status = `draft-ready`,
  sets `Contains VERIFY Markers`, sets `Missing Terms`
- **Admin site interaction:** Set status to `brief-ready` to trigger.
  No webhook needed — polling handles it.

### WF2 — Revision
- **Trigger:** Airtable status = `needs-revision` (polls every 5 min)
- **What it does:** Pre-checks revision notes exist and revision count ≤ 3.
  Calls Claude with current draft + revision notes + missing terms.
  Saves new draft, increments Revision Number, sets status = `draft-ready`.
  On pre-check failure: sets status = `error`, logs reason.
- **Admin site interaction:** Write revision notes to `Revision Notes` field,
  then set status to `needs-revision`. No webhook needed.

### WF3 — NeuronWriter Brief (semi-automated)
- **Trigger:** Airtable status = `nlp-pending` (polls every 5 min)
- **What it does:** Creates NeuronWriter analysis for the keyword, waits
  2 minutes, fetches NLP terms + word count, writes to NLP Brief fields,
  sets status = `brief-ready`
- **Admin site interaction:** Set status to `nlp-pending` to trigger.
  Article then flows automatically through to WF1.

### WF4 — NeuronWriter Import (manual batch)
- **Trigger:** Manual trigger in n8n (one-shot batch import)
- **What it does:** Reads NeuronWriter queries tagged 'import', creates
  Airtable records with NLP data, sets status = `nlp-ready`
- **Admin site interaction:** Not needed — run directly in n8n.

### WF5 — WordPress Publishing
- **Trigger:** Airtable status = `image-approved` (polls every 5 min)
- **What it does:** Generates SEO metadata via Claude, uploads selected
  image to WordPress Media API, creates WordPress draft post, stores
  WP Post URL + WP Post ID, sets status = `wp-draft`
- **Admin site interaction:** Set `Selected Image` field and set status
  to `image-approved`. WF5 fires automatically within 5 minutes.

## 6. Two user roles

The admin site has two roles with different access levels.

### Editor (Marko)
Full access to everything:
- Pipeline dashboard — all articles, all statuses
- Draft review — read rendered HTML, approve, request revision, send to enrichment
- NLP term coverage view
- Revision notes input
- Image brief writing
- Image approval — see all candidates, select, approve & trigger WF5
- Prompt editor (Templates table)
- Pipeline health metrics
- Error log viewer
- All status override controls

### Image Contributor (student)
Restricted view — can only see and do the following:
- List of articles with status `awaiting-images` (their queue)
- For each article: title, voice, image brief written by editor
- Upload up to 5 image candidates per article
- Add source/license notes
- Submit — sets status to `images-submitted`

The contributor cannot:
- See draft content
- Change any status other than the submit action above
- See NLP data, revision history, or pipeline health
- Access any other article (only `awaiting-images` ones)

**Auth approach:** Simple password-based. Two hardcoded credentials in
environment variables. No database, no OAuth. This is a single-operator
internal tool.

```
EDITOR_PASSWORD=...
CONTRIBUTOR_PASSWORD=...
SESSION_SECRET=...
```

Use iron-session or next-auth with credentials provider.

## 9. Key UI screens — what each one does

### Dashboard (`/dashboard`)
- Attention queue: articles that need the editor's action right now
  (statuses: draft-ready, images-submitted, error)
- Pipeline columns: article counts by status group
- Quick-access buttons to jump to the relevant article

### Article detail (`/articles/[id]`)
Shows only fields relevant to the article's current status.

**Tabs:**
- Draft — rendered HTML preview, VERIFY marker highlights, approve/revision actions
- NLP — term coverage side by side with draft, missing terms badge
- History — revision history from Revision History table

**Draft tab — left panel (rendered draft):**
- Render `Current Draft HTML` as actual HTML, not raw markup
- Highlight any `[VERIFY: ...]` tags in amber so they stand out immediately
- Word count indicator below the draft: actual word count vs target, color-coded:
  - Green: within 10% of target
  - Amber: 10–25% off target
  - Red: more than 25% off target
  - Example: `1,947 words — target 2,000 ✓`

**Draft tab — right panel (status indicators + actions):**

Missing Terms badge:
- Shows count of NLP terms absent from the draft: e.g. `4 missing terms`
- Amber badge if any missing, green badge if all present
- Expandable — click to see the full list of missing terms
- Source: `Missing Terms` field set by WF1/WF2, or compute live from
  `NLP Brief — Body Basic` + `NLP Brief — Body Extended` vs `Current Draft Plain Text`

Voice mismatch alert:
- Scan the plain text for voice-specific language patterns and flag if they
  contradict the article's Voice field
- Rules to implement:
  - Voice = `couples` but text contains "solo", "travelling alone", "by yourself" → flag
  - Voice = `solo` but text contains "your partner", "as a couple", "romantic" → flag
  - Voice = `family` but text contains "adults only", "no children" → flag
  - Voice = `active-hiking` but text contains no hiking/trail/elevation language → flag
- Display as a red alert banner above the action buttons if triggered:
  `⚠ Voice mismatch detected — article contains [solo] language but Voice is set to couples`
- Advisory only — does not block approval

Revision Notes textarea:
- Editable long-text field, pre-filled with any existing content from
  `Revision Notes` Airtable field (do not show empty placeholder if notes exist)
- Auto-saves on blur (PATCH to Airtable)
- Required to be non-empty before "Send to Revision" button is enabled

Action buttons (shown based on current status):
- `draft-ready`:
  - **Approve** → sets Status = `approved` (content approved, moves to image stage)
  - **Send to Revision** → writes Revision Notes + sets Status = `needs-revision`
    (disabled if Revision Notes is empty)
  - **Send to Enrichment** → sets Status = `needs-enrichment`
- `needs-revision` → show progress indicator "Revision in progress…", disable actions
- `approved` → show "Write image brief" textarea + "Send to image queue" button
  which sets Status = `awaiting-images`
- `error` → show Error Log content + "Reset to brief-ready" / "Reset to approved"
  manual override buttons

**Status-appropriate action buttons summary:**
- `draft-ready` → Approve (`approved`) / Send to Revision (`needs-revision`) / Send to Enrichment (`needs-enrichment`)
- `needs-revision` → waiting indicator (n8n handles automatically)
- `approved` → Write image brief + set to `awaiting-images`
- `error` → show Error Log, manual status reset controls

### Image approval (`/images/[id]`) — editor only
- Image brief display (what the editor asked for)
- 3–5 candidate images in a grid
- For each: hero crop preview (16:9) + thumbnail crop preview (1:1)
- Source notes from contributor
- Click to select, then Approve & publish button
- Approve & publish action: sets `Selected Image` + sets status to
  `image-approved` → WF5 fires within 5 minutes

### Image upload (`/upload`) — contributor only
- List of articles with status `awaiting-images`
- For each: title, voice, image brief
- Drag-and-drop upload (up to 5 files)
- Source notes textarea
- Submit button: uploads images to Airtable, sets status to `images-submitted`

### Prompts (`/prompts`) — editor only
- List of all templates from Templates table
- Click to view/edit prompt text
- Save as new version (creates new row, sets Active = true, deactivates old)
- Version history visible

### Health (`/health`) — editor only
- Articles processed this month
- Average revision cycles
- Error count
- Status distribution bar chart
- API cost estimate (based on token counts if available)

## 10. Airtable API patterns used by the admin site

### Fetch articles needing attention
```
GET /v0/{baseId}/{tableId}
  ?filterByFormula=OR({Status}='draft-ready',{Status}='images-submitted',{Status}='error')
  &sort[0][field]=Created
  &sort[0][direction]=asc
```

### Fetch single article
```
GET /v0/{baseId}/{tableId}/{recordId}
```

### Update article status
```
PATCH /v0/{baseId}/{tableId}/{recordId}
Body: { "fields": { "Status": "needs-revision", "Revision Notes": "..." } }
```

### Upload image to Airtable attachment field
Airtable does not accept binary uploads directly. The pattern is:
1. Upload the file to a temporary public URL (Vercel Blob or similar)
2. PATCH the Airtable record with the URL in the attachment field:
```
PATCH /v0/{baseId}/{tableId}/{recordId}
Body: {
  "fields": {
    "Image Candidates": [{ "url": "https://..." }]
  }
}
```
Airtable fetches and stores the file from the URL. The temp URL can
then be discarded.

### Fetch revision history for an article
```
GET /v0/{baseId}/Revision History
  ?filterByFormula={Article}='{recordId}'
  &sort[0][field]=Revision Number
  &sort[0][direction]=asc
```

## 11. NLP term coverage logic

The NLP brief tab shows which terms from the brief are present in the draft.
This runs client-side — no API call needed.

```typescript
function getTermCoverage(plainText: string, bodyBasic: string, bodyExtended: string) {
  const text = plainText.toLowerCase();
  const parseTerms = (raw: string) =>
    raw.split('\n')
       .map(t => t.replace(/:\s*\d+-?\d*x?/i, '').trim().toLowerCase())
       .filter(t => t.length > 2);

  const basic = parseTerms(bodyBasic);
  const extended = parseTerms(bodyExtended);

  return {
    basic: basic.map(term => ({ term, present: text.includes(term) })),
    extended: extended.map(term => ({ term, present: text.includes(term) })),
  };
}
```

### Voice mismatch detection

Run this client-side on the plain text whenever the draft tab loads.

```typescript
const VOICE_SIGNALS: Record<string, { own: string[], foreign: string[] }> = {
  couples: {
    own: ['partner', 'together', 'romantic', 'couple', 'honeymoon'],
    foreign: ['solo', 'travelling alone', 'by yourself', 'single traveller'],
  },
  solo: {
    own: ['solo', 'travelling alone', 'independent', 'by yourself'],
    foreign: ['your partner', 'as a couple', 'romantic getaway', 'honeymoon'],
  },
  family: {
    own: ['children', 'kids', 'family', 'little ones'],
    foreign: ['adults only', 'no children', 'child-free'],
  },
  friends: {
    own: ['group', 'friends', 'crew', 'squad'],
    foreign: [],
  },
  'active-hiking': {
    own: ['trail', 'hike', 'elevation', 'summit', 'trek', 'km', 'metres'],
    foreign: [],
  },
};

function detectVoiceMismatch(plainText: string, voice: string): string | null {
  const text = plainText.toLowerCase();
  const signals = VOICE_SIGNALS[voice];
  if (!signals) return null;
  const foreignFound = signals.foreign.filter(term => text.includes(term));
  if (foreignFound.length > 0) {
    return `Voice mismatch — article contains "${foreignFound[0]}" language but Voice is set to ${voice}`;
  }
  if (signals.own.length > 0) {
    const ownFound = signals.own.some(term => text.includes(term));
    if (!ownFound && voice === 'active-hiking') {
      return `Voice mismatch — no hiking/trail language found but Voice is set to active-hiking`;
    }
  }
  return null;
}
```

## 12. UI design reference

The mockup built in Claude.ai shows the intended design. Key design decisions:

- Clean, flat, minimal — similar to Linear or Notion
- Sidebar navigation with colored status dots
- Article cards show title, voice, revision number, and any alerts
- Status badges use semantic colors:
  - draft-ready → amber
  - needs-revision → red
  - images-submitted → blue
  - approved → green
  - published / wp-draft → gray
- VERIFY markers highlighted in amber within the rendered draft
- NLP terms: green = present, red = missing
- Image candidates in a 3-column grid with crop preview boxes
- Two-column layout on article detail: main content + sidebar panel

The site should feel native to the editor's workflow — every screen
answers the question "what do I do next?" rather than showing all
possible data at once.

## 13. What NOT to build

- Do not replicate n8n logic in the admin site. The site only reads/writes
  Airtable and triggers status changes. n8n does all automation.
- Do not build a prompt testing interface in v1 — the prompt editor
  (Templates table) is enough.
- Do not add user management — two hardcoded roles is sufficient.
- Do not connect directly to the Claude API from the admin site —
  all Claude calls go through n8n workflows.
- Do not auto-publish to WordPress — always create as draft, never publish.
- Do not build WF3/WF4 management UI in v1 — the NeuronWriter brief
  workflow is managed directly in n8n and Airtable for now.

## 14. Build order (recommended)

Build in this sequence so you have something working at each step:

1. **Auth** — login page, two roles, session middleware, protect all routes
2. **Airtable client** — `lib/airtable.ts` with typed fetch helpers
3. **Dashboard** — attention queue + pipeline column counts
4. **Article list** — filterable by status
5. **Article detail** — draft tab first (rendered HTML, approve/revision actions)
6. **Article detail** — NLP tab (term coverage)
7. **Image approval** — editor candidate selection + approve trigger
8. **Image upload** — contributor upload zone + submit
9. **Article detail** — History tab
10. **Prompts editor** — Templates table CRUD
11. **Health metrics** — pipeline stats

Deploy to Vercel after step 3 and keep deploying as you go.
